# Лабораторная работа 1 (VectorSum)
## Сумма элементов вектора на CPU и GPU (CUDA)

### Постановка задачи
Требуется реализовать алгоритм сложения элементов вектора двумя способами:
- на **CPU**;
- на **GPU** с использованием **CUDA**.

Входные данные: вектор размером от **1 000** до **1 000 000** элементов.  
Выходные данные: сумма элементов вектора и время вычисления.

Дополнительно требуется провести серию экспериментов для векторов разных размеров, сравнить время работы CPU и GPU и вычислить ускорение.

---

## Цель работы
Исследовать эффективность параллельного суммирования элементов вектора на графическом процессоре и сравнить её с последовательной реализацией на центральном процессоре.

---

## Использованные технологии
- **Язык:** C++ / CUDA
- **Среда выполнения:** Google Colab
- **Компилятор:** `nvcc`
- **GPU:** NVIDIA GPU, предоставляемый Google Colab

## Идея решения
В работе реализованы две версии алгоритма:

### 1. Суммирование на CPU
Функция `cpu_sum(...)` последовательно проходит по всем элементам вектора и накапливает сумму в одной переменной.

Это обычный последовательный алгоритм, который не использует параллелизм.

### 2. Суммирование на GPU
Для GPU использована **параллельная редукция**.

Суть подхода:
1. Исходный массив передаётся в память GPU.
2. CUDA-ядро запускает множество потоков.
3. Каждый поток обрабатывает часть массива и вычисляет локальную сумму.
4. Далее внутри блока выполняется редукция в `shared memory`.
5. Для каждого блока получается одна частичная сумма.
6. Процесс повторяется, пока не останется один итоговый элемент — общая сумма всего вектора.

---

## Что именно было распараллелено
Распараллелена операция суммирования элементов массива.

Почему это возможно:
- элементы массива можно обрабатывать независимо;
- разные потоки могут одновременно вычислять частичные суммы;
- затем эти частичные суммы объединяются в итоговый результат.

На GPU каждый поток складывает свою часть данных, а затем внутри блока результаты суммируются с помощью редукции в общей памяти (`shared memory`).

## Измерение времени
В этой работе время GPU измерялось **двумя способами**.

### 1. Полное время GPU
Измерялось время, включающее:
- выделение памяти на GPU (`cudaMalloc`);
- копирование данных из RAM в память GPU;
- выполнение CUDA-ядер;
- копирование результата обратно на CPU;
- освобождение памяти (`cudaFree`).

Это практический сценарий запуска программы целиком.

### 2. Чистое время ядра GPU
Измерялось только время выполнения CUDA-ядер, когда массив уже находился в памяти GPU.

Это позволяет оценить именно вычислительную эффективность параллельного алгоритма без накладных расходов на передачу данных и управление памятью.

### Измерение времени CPU
Для CPU измерялось только время последовательного суммирования элементов вектора.

---

## Особенности реализации
- Для генерации входных данных использовались случайные целые числа от **1** до **10**.
- Для каждого размера вектора выполнялось **20 повторов**, после чего вычислялось среднее время.
- Перед основными измерениями выполнялся **разогрев GPU**, чтобы исключить влияние первой инициализации CUDA-контекста.
- Для проверки корректности сравнивались результаты суммы, полученные на CPU и GPU.

## Анализ результатов
По таблице видно следующее:

### 1. Корректность вычислений
Во всех экспериментах суммы, полученные на CPU и GPU, совпадают. Это означает, что реализация работает корректно.

### 2. Полное время GPU больше времени CPU
Во всех проведённых экспериментах **полное время GPU** оказалось больше времени CPU.

Это ожидаемо, потому что в полное время входят дополнительные накладные расходы:
- выделение и освобождение памяти на GPU;
- копирование данных между CPU и GPU;
- запуск CUDA-ядер.

Для сравнительно небольших массивов эти накладные расходы оказываются больше выигрыша от параллельных вычислений.

### 3. Чистое время ядра GPU становится выгоднее на больших размерах
Если рассматривать только время работы CUDA-ядер, без передачи данных, то GPU начинает выигрывать примерно с размера **100 000** элементов.

Например:
- при **100 000** элементов ускорение по времени ядра составило **1.496x**;
- при **500 000** элементов — **1.520x**;
- при **1 000 000** элементов — **2.795x**.

Это показывает, что сам параллельный алгоритм на GPU действительно эффективен, но его преимущества проявляются в основном на более крупных объёмах данных.

### 4. Почти постоянное время ядра на малых размерах
Для размеров от 1 000 до 100 000 элементов время ядра GPU остаётся примерно на уровне **0.028–0.030 мс**.

Это тоже нормальный результат. На малых входных данных существенную часть времени занимает не сама арифметика, а накладные расходы на запуск ядра и синхронизацию.

## Вывод
В ходе лабораторной работы был реализован алгоритм суммирования элементов вектора:
- в последовательном варианте на CPU;
- в параллельном варианте на GPU с использованием CUDA.

Эксперименты показали, что:
- GPU-реализация корректно вычисляет сумму элементов вектора;
- при измерении полного времени GPU проигрывает CPU из-за накладных расходов на выделение памяти и передачу данных;
- при измерении только времени CUDA-ядер GPU начинает показывать ускорение на больших размерах входного массива;
- максимальное ускорение по времени ядра в проведённых экспериментах составило **2.795x** для массива размером **1 000 000** элементов.

Таким образом, параллельное суммирование на GPU целесообразно прежде всего для больших объёмов данных и в тех случаях, когда затраты на передачу данных можно уменьшить или скрыть.
