# Лабораторная работа 0 (MatMul)
## Перемножение матриц на CPU и GPU (CUDA)

### Описание задания
В рамках лабораторной работы реализовано перемножение двух квадратных матриц двумя способами:

- на **CPU**;
- на **GPU** с использованием **CUDA**.

Программа принимает на вход две матрицы размера от `100x100` до `2000x2000`, выполняет их перемножение, проверяет корректность результата GPU-версии относительно CPU-версии и измеряет время выполнения.

---

## Цель работы
Изучить применение технологии CUDA для ускорения вычислений на графическом процессоре на примере задачи перемножения матриц и сравнить производительность CPU- и GPU-реализаций.

## Описание реализации

### 1. Умножение матриц на CPU
CPU-версия реализована стандартным тройным циклом.  
Для каждой строки матрицы `A` и каждого столбца матрицы `B` вычисляется соответствующий элемент результирующей матрицы `C`.

Используемая формула:
$$
\[
C[i][j] = \sum_{k=0}^{N-1} A[i][k] \cdot B[k][j]
\]
$$

В коде используется порядок циклов `i-k-j`, который работает лучше с точки зрения кэширования памяти, чем классический `i-j-k`.

---

### 2. Умножение матриц на GPU
GPU-версия реализована с помощью CUDA-kernel.

Для распараллеливания использован следующий подход:

- каждый поток CUDA вычисляет **один элемент** результирующей матрицы `C`;
- блоки потоков организованы как двумерная сетка;
- для уменьшения числа обращений к глобальной памяти используется **shared memory**;
- матрицы обрабатываются по частям (тайлам) размером `16x16`.

---

## Что именно было распараллелено и почему
Распараллелено вычисление элементов результирующей матрицы.

Каждый элемент `C[i][j]` можно вычислять независимо от остальных, так как он зависит только от строки матрицы `A` и столбца матрицы `B`.  
Это делает задачу хорошо подходящей для параллельных вычислений на GPU.

Почему это эффективно:

- между вычислениями разных элементов нет зависимости;
- можно запускать тысячи потоков одновременно;
- использование разделяемой памяти (`shared memory`) снижает затраты на чтение данных из глобальной памяти видеокарты.

---

## Проверка корректности
Корректность GPU-реализации проверяется сравнением результата с CPU-версией.

Для этого вычисляется:

- максимальная абсолютная ошибка `max_abs_err`;
- максимальная относительная ошибка `max_rel_err`.

В итоговой версии критерием корректности выбран порог по абсолютной ошибке, так как при вычислениях в типе `float` относительная ошибка может быть большой для элементов, близких к нулю, даже если фактическое расхождение очень мало.

Используемый критерий:
```cpp
int correct = (max_abs_err < 1e-3) ? 1 : 0;
```

---

## Измерение времени
В программе измеряются три величины:

- `cpu_ms` — время работы CPU-версии;
- `gpu_kernel_ms` — время выполнения только CUDA-kernel;
- `gpu_total_ms` — полное время GPU-вычисления, включая копирование данных между host и device.

Также вычисляется ускорение:

- `speedup_kernel = cpu_ms / gpu_kernel_ms`
- `speedup_total = cpu_ms / gpu_total_ms`

## Анализ результатов
По результатам экспериментов можно сделать следующие выводы:

1. Для маленьких матриц (`100x100`) GPU проигрывает CPU по **полному времени выполнения**, так как накладные расходы на копирование данных между оперативной памятью и памятью видеокарты оказываются слишком заметными.

2. Начиная с размера `250x250`, GPU уже начинает превосходить CPU даже с учетом полной стоимости вычисления.

3. При увеличении размера матриц ускорение возрастает. Это связано с тем, что GPU лучше раскрывает свой параллельный потенциал на более крупных задачах.

4. Для матрицы `1000x1000` ускорение составило:
   - примерно **40.38 раза** по времени выполнения kernel;
   - примерно **15.33 раза** по полному времени выполнения.

5. Во всех экспериментах получен корректный результат (`Correct = YES`), а абсолютная ошибка остается очень малой, что подтверждает корректность GPU-реализации.
